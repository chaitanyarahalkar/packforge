# M2 implementation plan — Linux ELF x86-64 static executables

**Status: in progress; cold-start remediation planned.** The direct v2 runtime,
format, compatibility matrix, and native evidence campaign are implemented. The
2026-07-11 campaign passes correctness, reversibility, determinism, size, RSS,
loader, direct-mapping, and kernel gates, but median cold startup is 389.50% of
UPX. M2 is not complete merely when a direct loader works: the same stable
profile must pass every gate below against pinned UPX 5.2.0.

## Scope

M2 supports static, non-PIE, little-endian ELF64 `ET_EXEC` files on Linux x86-64.
PIE, `PT_INTERP`, `PT_DYNAMIC`, other architectures, and other operating systems
remain later milestones. Executable format v1 remains readable and reversible,
but it is an evidence-producing spike rather than the M2 release design.

## Baseline and performance diagnosis

Native workflow
[`29173099748`](https://github.com/chaitanyarahalkar/packforge/actions/runs/29173099748)
is the pre-M2 baseline. Correctness, reversibility, and determinism pass, but the
median Packforge/UPX size is 156.78% and median cold time is 135.65%. The v1
runtime is 10,888 bytes and incurs work UPX does not: it reads and hashes the
loader again, allocates separate payload and complete-original mappings, writes
the reconstructed file to a memfd, then asks the kernel to perform a second ELF
load through `execveat`.

`-12`, Zstandard 1.5.7 `-19`, and XZ Utils 5.8.2 LZMA-alone `-9e`. These are not
release results; they isolate whether a runtime implementation can plausibly pass
the size gate.

LZ4-HC keeps the current decoder but cannot close the size gap. The prior `ruzstd`
spike adds 59,936 bytes before the rest of the loader and is not viable. With a
23,500-byte total loader plus the fixed header, trailer, and actual 3/4-segment
manifest sizes, LZMA-alone is 98.2%, 98.3%, 104.8%, and 93.5% of UPX respectively.
The Rust fixture sets the hard feasibility boundary: the complete loader must be
at most 23,839 bytes to keep every fixture within 5% of UPX. M2 therefore uses a
23,500-byte go/no-go budget and targets a smaller result, rather than consuming
the old 32 KiB ceiling by default.

The admitted pure-Rust `lzma-sdk-rs` candidate and shared bounded decoder passed
the native gate in workflow
[`29174191514`](https://github.com/chaitanyarahalkar/packforge/actions/runs/29174191514).
The feature-retained loader is 14,776 bytes. Complete projected artifacts are
294,077, 294,002, 196,746, and 601,851 bytes: 95.19%, 95.17%, 100.28%, and
92.13% of UPX. The median is 95.18%, and all fixtures pass the 105% bound. This
admits the codec for v2 integration but is not a substitute for the final direct
runtime and cold-start benchmark.

## Cold-start remediation plan — 2026-07-12

The failed native campaign leaves 6,140 bytes below the 23,500-byte loader
ceiling, while the current runtime is compiled globally for minimum size with
`opt-level = "z"`. Warm execution alone is 23–35 ms for the C, C++, and Rust
fixtures and 68 ms for Go, so page-cache I/O cannot explain the failure. The
first optimization target is the bounded LZMA1 decoder and its generated code;
security checks are not optional optimization candidates.

Work proceeds in the following measured order:

1. Add a reproducible runtime build matrix for `z`, `s`, `1`, `2`, and `3` codegen.
   Record normalized loader size and identical decoder-vector results for every
   candidate. Reject any candidate above 23,500 bytes or with different output,
   framing, malformed-input behavior, ELF properties, or reproducibility.
2. Benchmark the admitted candidates on native Linux using the exact four
   release fixtures. Promote the smallest candidate whose median cold time beats
   UPX while every per-fixture size stays at or below 105% of UPX. Codegen is a
   release-profile decision, not an unrecorded local flag.
3. If no codegen candidate passes, add phase evidence for payload read/hash,
   decompression, segment population, and transfer preparation. Use an untimed
   diagnostic loader so measurement hooks cannot alter the released artifact.
4. Optimize only the dominant measured phase. Permitted next changes are bounded
   decoder hot-loop specialization, chunked match copies with overlap-safe
   semantics, and removal of redundant copying through a verified direct-output
   layout. Payload, original-image, manifest, and loader authentication; exact
   recovery; W^X; collision rejection; and fail-closed parsing remain required.
5. Run the full two-runner compatibility and release campaign after every
   admitted runtime change. Check in raw samples and reports from one commit,
   update the README and roadmap only on an overall pass, and retain failed
   campaigns as evidence.

Acceptance requires the strict M2 report—not a microbenchmark—to show an overall
pass. In particular, median cold startup must be strictly below UPX `--best` on
the pinned native campaign; the looser product-level size-win alternative does
not close this milestone.

### Current candidate evidence

The first matrix admitted selective `opt=3` codegen for the LZMA decoder and
`opt=2` for the compact BLAKE3 implementation while retaining size-oriented
codegen for the rest of the loader. The normalized candidate is 22,720 bytes,
has no runtime relocations, and leaves 780 bytes below the loader ceiling. Exact
Rust and Go fixture phase benchmarks reduced combined decode and authentication
work by roughly 47% compared with the released 17,360-byte size-only build; the
decoder `opt=3` candidate was consistently 1.6–3.6% faster than `opt=2` in those
measurements.

Because the host format and payload bytes are unchanged, applying the 5,360-byte
loader delta to the checked-in release report gives projected Packforge/UPX size
ratios of 97.77%, 97.75%, 104.33%, and 93.35% for C, C++, Rust, and Go. The
projected median is 97.76%, and the Rust fixture remains below the 105% ceiling.

Global speed codegen, the upstream pure-Rust BLAKE3 crate, chunked match copying,
and literal unrolling were rejected for loader-size, relocation, or measured
speed regressions.

Native workflow
[`29196951813`](https://github.com/chaitanyarahalkar/packforge/actions/runs/29196951813)
at commit `9c98f37ca38e572c5df706a5903743f78c4d9002` confirms the candidate's
correctness, reversibility, determinism, two-runner compatibility, direct mapping,
loader size, packed size, and RSS gates. Median packed size is 97.75% of UPX and
median RSS is 103.38%, but median cold startup is still 265.07% of UPX. M2
therefore remains open. The next native run records seven timestamped syscall-gap
samples per fixture for payload read, payload authentication, decompression and
original verification, fixed-address mapping, and transfer so further work is
based on the dominant measured phase.

### Decoder hot-path plan

The first timestamped traces attribute median C/C++ startup to roughly 18.7 ms
of LZMA decompression plus original verification, 2.1 ms of fixed-address mapping,
0.66 ms of payload authentication, and 0.15 ms of payload reads. Decoder work is
therefore the only phase large enough to close the remaining startup gap.

The next candidate separates the runtime-only unchecked hot path from the safe
host decoder while keeping one shared algorithm and error model:

1. Move the decoder body into one shared source module with safe indexing hooks
   for host verification and explicitly unchecked hooks for the freestanding
   runtime only.
2. Permit unchecked reads or writes only after the existing input-position,
   distance, dictionary, and output-end checks establish the exact valid range.
   Probability-table indices remain derived from bounded LZMA states and bit-tree
   depths; malformed streams must retain the same errors as the safe decoder.
3. Run identical encoder vectors, boundary cases, corrupt/truncated streams,
   differential fuzz targets, Clippy, and sanitizer-compatible host tests against
   both implementations. Any output or error mismatch rejects the candidate.
4. Admit the runtime candidate only if it remains relocation-free, reproducible,
   below 23,500 bytes, and materially reduces the native `decompress` phase. The
   complete four-fixture cold-start report remains the release decision.

The first implementation of this plan was rejected. Compiling the shared decoder
with unchecked runtime-only indexing grew the normalized loader from 22,720 to
23,024 bytes. Its exact-fixture host medians were 6.34 ms for Rust and 20.92 ms
for Go, versus approximately 6.31 ms and 20.76 ms for the safe candidate. Bounds
checks are therefore not the dominant cost, and the unchecked crate was removed.

### Guarded range-decoder plan

The next candidate targets the per-bit fallible control flow and probability
layout, using the public-domain 7-Zip 26.02 decoder as a performance oracle. It
does not import a general allocator or weaken the fixed Packforge format contract:

1. Benchmark the pinned 7-Zip C decoder on the exact Rust and Go payloads, and
   compile its x86-64 optimized core separately to record the attainable decoder
   speed and text-size bounds before changing the runtime.
2. Replace per-bit `Result` propagation with a guarded core. Enter the no-fail
   range loop only when at least `LZMA_REQUIRED_INPUT_MAX` bytes remain; process
   the final window through a checked tail buffer so truncated input still fails
   without an out-of-bounds read.
3. Use the reference decoder's flat probability layout and fixed-profile bit-tree
   specialization only where differential traces prove identical state updates.
   Keep dictionary, match-distance, exact-output, end-marker, and trailing-data
   validation outside the no-fail primitive and fail closed on every mismatch.
4. Differential-test valid vectors, every truncation boundary, deterministic bit
   mutations, and the four release payloads against both the existing safe Rust
   decoder and the pinned 7-Zip oracle. Run Clippy and sanitizer-compatible host
   tests before runtime admission.
5. Admit the candidate only if it is relocation-free, reproducible, no larger
   than 23,500 bytes, and cuts the native decompression phase enough to make a
   full four-fixture release campaign credible. Otherwise reject it and revisit
   the codec/format tradeoff rather than accumulating unmeasured unsafe code.

The reference feasibility bound rejects this direction before runtime
integration. On the exact local Rust and Go payloads, 7-Zip 26.02's `-O3` C
decoder measured 5.53 ms and 18.40 ms, only about 12% faster than the current
6.31 ms and 20.76 ms Rust medians. Encoder levels 0, 1, 3, 5, 7, and 9 changed
decode time by less than roughly 3%. Even a perfect integration of that C core
cannot close the approximately fourfold native phase gap, so no C/assembly or
guarded unsafe decoder was added to the runtime.

### BCJ independent-stream feasibility plan

LZMA's serial range decoder is now the measured constraint. The next format
candidate tests whether a reversible x86 BCJ transform creates enough size
headroom to split the image into independently decodable streams:

1. Add a host-only spike that applies the public-domain x86 BCJ transform, splits
   the transformed image at deterministic output offsets, encodes each chunk with
   the existing fixed LZMA profile, and verifies exact inverse reconstruction.
2. Evaluate one, two, three, and four streams on the exact four release fixtures.
   Include the chunk table and the remaining 768-byte runtime reserve in projected
   artifacts; reject any layout above 105% of UPX for one fixture. This is a
   best-case feasibility bound because the 23,500-byte loader limit allows no
   larger increase.
3. Admit parallel runtime work only if at least four balanced-output streams pass
   the size gate. Two streams cannot supply the measured fourfold phase reduction,
   and uneven streams are limited by the slowest worker.
4. If admitted, specify a bounded freestanding worker contract before coding:
   fixed anonymous stacks with guard pages, `clone` workers sharing only the
   output mapping and immutable payload, futex completion, per-chunk decode and
   digest results, deterministic failure propagation, complete worker teardown,
   inverse BCJ, final whole-image authentication, and restoration of target-visible
   signal/process state before transfer.
5. Differential-test serial and parallel reconstruction plus worker failures,
   then run the same relocation, reproducibility, size, RSS, syscall, behavior,
   and cold-start gates. Reject the parallel design if process semantics or the
   strict release report differ from the original.

Initial local bounds justify only the host feasibility gate: BCJ plus one LZMA
stream measured 180,158 bytes for Rust and 562,653 bytes for Go; four equal-output
streams measured 185,950 and 570,418 bytes. The CI corpus and projected loader
delta decide whether those increases fit the actual per-fixture ceilings.

Native workflow
[`29198061091`](https://github.com/chaitanyarahalkar/packforge/actions/runs/29198061091)
rejects the standalone parallel layout. Four streams pass the projected C, C++,
and Go size ceilings, but the Rust fixture is 207,987 bytes against a 206,005-byte
limit. Three streams also miss Rust by 271 bytes; two streams pass every fixture
but cannot supply the measured phase reduction. Runtime compatibility remains
green on Ubuntu 22.04 and 24.04, and the unchanged release candidate still passes
every M2 gate except cold start at 264.28% of UPX.

### Direct-output mapping plan

Before reconsidering parallel streams, remove the measured segment-copy phase and
its staging mapping without changing codec bytes or executable format:

1. Derive a checked primary file bias from the first manifest segment and reserve
   the union of the complete original file plus every `p_memsz` range as one
   bounded virtual span. ELF congruence permits later writable segments to use a
   different page bias, so do not require every segment bias to be identical.
2. Reserve that span once with `MAP_FIXED_NOREPLACE` and writable, non-executable
   anonymous pages. Decompress the original directly at `load_bias + file_offset`,
   so ELF bytes already occupy their final addresses and BSS begins zeroed.
3. Authenticate and re-parse the direct output before making anything executable.
   Leave segments at the primary bias in place; relocate only exceptional segment
   file ranges with overlap-safe `memmove`. Require canonical file and mapping
   order so one forward/backward adjacency pass rejects cross-segment source
   clobbering; then zero BSS, unmap gaps, and apply final protections in order.
4. Add host manifest regressions for invalid primary bias, cross-segment source
   clobbering, and oversized spans;
   run both native compatibility families, syscall traces, reproducibility, size,
   RSS, and phase timings. Any mapping or behavior mismatch rejects the layout.
5. Admit the change only if the normalized loader stays at or below 23,500 bytes
   and the native `map_segments` phase materially falls. Re-run the four-stream
   size model against the smaller admitted loader before planning worker code.

The admitted hybrid implementation in workflow
[`29198877462`](https://github.com/chaitanyarahalkar/packforge/actions/runs/29198877462)
is 23,200 bytes, reproducible across Linux x86-64 and macOS ARM64, relocation-free,
and green on both compatibility runners. It removes the full segment copy and
relocates only page-shifted file ranges. Median cold startup improves from 264.28%
to 253.30% of UPX, so M2 remains open.

The syscall-only phase parser requires one contract amendment for this layout.
Legacy traces retain their two-staging-map boundaries. Direct-output traces use
the fixed output mapping as the decode start and the first final `mprotect` as the
end of the combined decompression, original verification, exceptional relocation,
and BSS-zeroing interval. `map_segments` then covers final protection/unmapping
only. No finer attribution is claimed without an explicit diagnostic build.

### Native x86-64 assembly oracle plan

The corrected direct-output phases still leave approximately 18.9 ms of combined
decode work for C/C++ and 12.2 ms for Rust. Before designing workers, establish
whether the official optimized x86-64 decoder can make four streams sufficient:

1. Pin 7-Zip 26.02 commit `f9d78aff31a5f2521ae7ddbdc97c4a8855808959`
   and Asmc commit `4b669147521b277b9e050922e7c97cb8aa608f45` in a
   native-only diagnostic script. Assemble `LzmaDecOpt.asm`, compile its coupled
   C driver, and record object/text size and 21-run medians.
2. Extract the exact raw payload and five property bytes from each packed release
   fixture; require byte-identical output from the oracle before timing it. Do not
   substitute a synthetic 7-Zip benchmark or a different encoder stream.
3. Combine measured whole-stream oracle time with the admitted direct-output
   non-decode phases. Admit a four-worker design only if the projected C, C++, and
   Rust cold times are below their same-run UPX medians with explicit allowance
   for worker creation/synchronization.
4. Treat size as a replacement bound, not just an object-file observation. The
   assembly core, worker trampoline, BCJ inverse, chunk metadata, and smaller
   decoder codegen must keep the complete loader at or below 23,500 bytes and the
   four-stream Rust artifact at or below 105% of UPX.
5. If both speed and size bounds pass, write the worker/process-state contract
   before implementation. Otherwise reject parallel LZMA and reopen the codec or
   release-gate contract rather than adding concurrency that cannot close M2.

Workflow
[`29199174024`](https://github.com/chaitanyarahalkar/packforge/actions/runs/29199174024)
passes the complete unchanged M2 campaign and validates the oracle against every
release fixture. Whole-stream assembly medians are 10.91 ms for C, 10.90 ms for
C++, 7.10 ms for Rust, and 23.05 ms for Go. Four-way decoding therefore leaves
approximately 1.3--1.7 ms of worker overhead while still projecting below each
same-run UPX cold median. The assembly object is 5,477 bytes; replacing the Rust
decoder rather than retaining both implementations gives a plausible complete
loader bound. The four-stream design is admitted for implementation, not for
release: the native campaign below remains authoritative.

### Four-stream assembly runtime implementation plan

Freeze the codec and worker contracts before changing production artifacts:

1. Add wrapper codec 4 without changing codec 3. Codec 4 stores a canonical
   four-entry table containing each compressed offset/length and exact decoded
   range. Apply the reversible x86 BCJ transform to the complete original before
   splitting it. Choose boundaries deterministically from a bounded search whose
   largest decoded range is at most 28% of the original; minimize total encoded
   bytes, then maximum encoded bytes, then lexicographic boundaries. Encode all
   streams with the existing fixed raw-LZMA1 properties. Existing codec-3 files
   remain host-verifiable and unpackable, and their embedded old loaders remain
   runnable; new balanced packs select codec 4 only after all codec-4 gates pass.
2. Vendor the exact public-domain `LzmaDecOpt.asm` source from pinned 7-Zip 26.02
   commit `f9d78aff31a5f2521ae7ddbdc97c4a8855808959`, its license/provenance note,
   and the reproducible x86-64 ELF object assembled with pinned Asmc commit
   `4b669147521b277b9e050922e7c97cb8aa608f45`. Linux CI rebuilds the object and
   byte-compares it with the checked-in artifact; cross-host runtime builds link
   only that verified object. Reject dynamic libraries, host-native assembly, or
   silently different code generation.
3. Replace the Rust LZMA decoder in the codec-4 loader with a minimal fixed-profile
   assembly driver. Mirror the required `CLzmaDec` layout with compile-time offset
   assertions, initialize the fixed properties and probability table exactly as
   the reference C driver does, and give every compressed range at least
   `LZMA_REQUIRED_INPUT_MAX + 1` readable zero-padding bytes. Call
   `LzmaDec_DecodeReal_3` with exact input/output bounds and reject bad return
   state, early input/output completion, impossible distances, or leftover data.
   First prove one-stream byte identity and malformed-stream rejection before
   enabling the four-stream host format.
4. Decode four disjoint ranges with three raw `clone` workers plus the main
   thread. Allocate fixed, bounded anonymous stacks with guard pages; share only
   immutable compressed input, disjoint writable output, and a compact completion
   record. Workers perform no allocation, TLS, signal, descriptor, filesystem, or
   process-state mutation. They publish checked decode status with release
   ordering, wake the main thread with `futex`, and terminate with `SYS_exit`.
   The main thread handles clone failure deterministically, waits for and tears
   down every created worker before any error or target transfer, and reports the
   first chunk error in table order. No worker may survive inverse BCJ or mapping
   protection changes.
5. Preserve authentication boundaries. The wrapper payload digest covers the
   complete codec-4 table and compressed streams. After all chunks finish, apply
   inverse BCJ once over the contiguous output, verify the original whole-file
   digest, re-parse the ELF and compare it with the manifest, then perform the
   admitted exceptional-range relocation, BSS zeroing, final protections, auxv
   rewrite, and direct transfer. Parallel success must be observationally
   identical to the serial codec-3 path.
6. Enforce size gates after each stage. The serial assembly loader must be at most
   21,506 bytes with equal streams, or at most the recomputed Rust-fixture ceiling
   from the deterministic optimized split; never exceed the absolute 23,500-byte
   loader cap. Add BCJ, table parsing, clone/futex code, and checked error paths
   incrementally, measuring the final normalized loader and every complete
   fixture after each step. Delete codec-4 runtime code if the Rust artifact
   cannot remain at most 105% of same-run UPX.
7. Differential-test codec 4 against codec 3 and the original across the corpus,
   randomized boundaries, truncated/padded streams, corrupt tables, worker clone
   failures, worker decode failures, and partial startup. Trace that workers use
   only the declared memory/syscall contract and that all mappings are non-W+X.
   Then run formatting, clippy, unit/fuzz tests, deterministic cross-host builds,
   Ubuntu 22.04/24.04 compatibility, syscall/process semantics, size, RSS, exact
   unpacking, and the full 21-warm/seven-cold benchmark campaign. M2 closes only
   if every original release gate is green in one published native workflow.

The serial admission checkpoint in workflow
[`29200113478`](https://github.com/chaitanyarahalkar/packforge/actions/runs/29200113478)
passes native codec-4 execution, host verification, and byte-identical unpacking
with the pinned assembly object. The relocation-free loader is 21,744 bytes with
SHA-256 `2a15876c5c8206461de227e2b0eacfc64c378ecdee67f601ac61222fdd04ca62`,
which fits both the absolute loader cap and the optimized Rust-fixture bound.
The other jobs in that integration workflow intentionally compared the changed
source with the prior codec-3 artifact and failed reproducibility; the serial
assembly artifact replaces that prior artifact before the next full campaign.

## Deliverables

1. Phase-attributed measurement contract
   - preserve benchmark report v1 and its raw baseline unchanged;
   - add a versioned M2 report containing codec, decoder memory, loader bytes,
     syscall counts, warm/cold time, RSS, and per-phase development timings;
   - retain 21 warm samples, seven page-cache-reset cold samples, and the exact
     environment/toolchain metadata for Packforge and pinned UPX.
2. Bounded compact-codec feasibility gate
   - implement a deterministic raw LZMA1 host encoder with fixed, documented
     properties and no platform library dependency;
   - prototype a freestanding decoder that rejects malformed ranges, distance
     underflow, output overflow, trailing data, and dictionaries above its cap;
   - require byte-identical decoding for the full corpus and fuzz-generated data;
   - stop integration if the complete reproducible loader exceeds 23,500 bytes or
     projected corpus artifacts fail the per-fixture 105% size bound.
3. Executable format v2 and manifest integration
   - embed manifest v0, codec properties, exact range digests, and checked offsets
     in an independently versioned wrapper;
   - derive manifest records from every validated `PT_LOAD` header and compare
     them with re-parsed original metadata during host verification;
   - preserve host inspect/verify/unpack support for executable v1 and exact
     recovery of the complete original file from v2.
4. Position-independent direct runtime
   - build the freestanding loader as static `ET_DYN` with no interpreter,
     dynamic dependencies, writable-executable segment, or runtime relocation;
   - map the target `PT_LOAD` ranges at their declared virtual addresses with
     checked page arithmetic and collision rejection;
   - decompress into final or bounded staging mappings, zero `p_memsz - p_filesz`,
     apply final ELF protections only after writes, and never create W+X memory;
   - rewrite `AT_PHDR`, `AT_PHENT`, `AT_PHNUM`, `AT_ENTRY`, `AT_BASE`, and other
     target-owned auxiliary-vector values while preserving kernel-owned entries;
   - transfer control with the original stack, arguments, environment, random
     bytes, inherited descriptors, signal state, and static-entry register state;
   - remove the v1 `memfd_create`, full-file write, seal, and second `execve`
     launch path from v2 execution.
5. Compatibility and adversarial matrix
   - run C, C++, Rust/musl, and pure-Go fixtures plus cwd, descriptors, output,
     file effects, exit status, signals, `/proc/self/exe`, auxv, TLS, large BSS,
     sparse segments, mixed protections, and page-boundary cases;
   - reject overlapping segments, mapping collisions, unsupported flags,
     malformed manifests, corrupt codec streams, and impossible auxv layouts;
   - fuzz the host parser, manifest, codec decoder, and runtime primitives, then
     preserve every discovered failure as a minimized regression case.
6. Native release campaign
   - test at least two pinned native Linux kernel/runner families with ASLR on;
   - prove deterministic artifacts and byte-identical recovery across Linux
     x86-64 and macOS ARM64 hosts;
   - run all artifacts under memory and process limits, audit every lockfile, and
     publish raw benchmark samples plus the gate evaluation.

## Implementation sequence and gates

1. Add phase attribution and a v2 benchmark schema without changing the runtime.
2. Build the LZMA1 encoder/decoder spike in isolation. Admit it only after the
   23,500-byte complete-loader and projected-size gates pass.
3. Specify executable format v2, integrate manifest production/verification, and
   add host-only round trips before any native control transfer.
4. Produce the relocation-free `ET_DYN` loader and map a synthetic one-segment
   fixture in process. Prove W^X, auxv rewriting, and collision failure first;
   follow the checked sequence in [`M2_DIRECT_RUNTIME.md`](M2_DIRECT_RUNTIME.md).
5. Expand mapping to the full four-language and process-semantics corpus. A syscall
   trace must show no post-start `memfd_create`, full-original write, or `execve`.
6. Integrate the admitted compact codec, then optimize decoder copy count, hashing,
   page layout, and read strategy using phase data rather than aggregate timing.
7. Run the kernel, fuzz, audit, cross-host, and benchmark campaigns. Repeat
   optimization until every release gate below passes; do not relabel a failed
   candidate as M2 complete.

## Exit evidence

- Every supported fixture and behavior oracle matches the original, unpack is
  byte-identical, and repeated packing is byte-identical.
- The complete loader is at most 23,500 bytes and has no W+X segment, interpreter,
  dynamic dependency, or runtime relocation.
- The released `balanced` profile has median packed size strictly below UPX
  `--best`, every fixture is at most 105% of UPX, and median cold-start time is
  strictly below UPX on the same pinned run.
- Median peak RSS is at most 110% of UPX, and declared codec memory is enforced
  before allocation.
- Native traces prove direct transfer without memfd reconstruction or a second
  exec, and the pinned kernel matrix is green with ASLR enabled.
- Parser, manifest, codec, and runtime fuzz campaigns have no unresolved crash,
  timeout, unbounded allocation, or semantic mismatch; dependency-audit findings
  remain zero.
- The checked-in report, raw samples, README comparison, and roadmap all link to
  the same successful native campaign before M2 is marked complete.

M3 planning starts only after this audit passes. A size-only or startup-only win
does not satisfy the M2 goal.
