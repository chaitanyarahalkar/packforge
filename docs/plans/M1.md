# M1 completion plan — reversible container

M1 promotes the existing `.pfg` implementation from a working vertical slice to
a closed, portable recovery-container milestone. It does not change native
runtime behavior; executable optimization begins only after this milestone's
reversibility and malformed-input evidence is complete.

## Current evidence

- Container v1 has a fixed 192-byte header, hard input/output limits, BLAKE3
  integrity, LZ4/Zstandard codecs, deterministic profiles, and exact trailing-data
  rejection.
- Host pack, inspect, verify, and unpack APIs plus atomic no-clobber file writes
  are implemented.
- Unit and CLI tests cover basic round trips, corruption, determinism, size limits,
  and unsupported ELF features.
- The existing Linux container smoke test covers one static C executable; it is
  not yet the complete supported corpus or a cross-host reproducibility proof.

## Deliverables

1. Complete supported-corpus matrix
   - C, C++, Rust/musl, and pure-Go static non-PIE executables;
   - `fast`, `balanced`, `small`, and `auto` profiles;
   - pack twice, inspect, verify, unpack, and compare exact bytes and mode;
   - record codec selection and artifact digest as deterministic golden metadata.
2. Malformed and resource-bound corpus
   - every header field, enumeration, reserved range, digest, checked length, and
     trailing-data path;
   - truncated LZ4 and Zstandard inputs, oversized declared output, and hostile
     high-window Zstandard frames;
   - minimized regression fixtures for every discovered parser/decompressor bug.
3. Stable automation contract
   - JSON Schema for container inspect/verify output with golden examples;
   - stable exit-code and diagnostic catalog for unsupported versus corrupt input;
   - schema compatibility tests that reject accidental field removal or semantic
     reuse.
4. Cross-host determinism
   - identical container bytes on native Linux x86-64 and macOS ARM64 using the
     pinned Rust/toolchain and fixture sources;
   - CI uploads and compares artifact digest manifests rather than relying on
     separate green jobs.
5. Release campaign
   - parser/decompressor fuzzing over the complete container surface;
   - full corpus round-trip campaign under memory limits;
   - dependency audit and documented unresolved-finding count of zero.

## Implementation sequence

1. Expand the Linux smoke runner to the four-language/four-profile matrix and
   emit a deterministic digest manifest.
2. Add golden JSON report schemas/examples and compatibility tests.
3. Add the malformed/resource-bound regression matrix and decompressor limits.
4. Add macOS artifact generation plus cross-job digest comparison.
5. Execute the release campaign and audit every M1 exit criterion.

## Exit evidence

- All 16 language/profile combinations round-trip byte-identically and repeated
  packing is byte-identical.
- Linux and macOS produce the same digest manifest for identical source artifacts.
- All malformed/resource-bound cases fail with the documented class and without
  unbounded allocation.
- JSON output validates against the checked-in schema and golden examples.
- The linked fuzz/release campaigns contain no unresolved finding.
- `ROADMAP.md` changes M1 to complete only after each item is directly evidenced.

M2 planning and performance implementation begin only after this audit passes.
