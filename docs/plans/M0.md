# M0 completion plan — measurement and format contract

M0 exists to make performance and correctness claims reproducible before runtime
work is optimized against them. It closes only when another Linux x86-64 machine
can rebuild the corpus, reproduce a versioned report, and obtain an explicit
pass/fail result against the pinned UPX baseline.

## Scope

M0 measures the current static, non-PIE Linux x86-64 compatibility tier. It does
not expand executable compatibility or change the runtime. Later milestones may
add corpus families, but they must not rewrite M0 fixture identities or result
semantics.

## Deliverables

1. `benchmarks/corpus-v1.json`
   - stable fixture IDs, source digests, licenses, build recipes, target facts,
     and behavioral oracle;
   - deterministic C, C++, Rust/musl, and pure-Go entries.
2. `benchmarks/schema/benchmark-report-v1.schema.json`
   - environment, toolchain, artifact, correctness, warm-time, cold-time, RSS,
     and aggregate-gate fields;
   - integer base units only: bytes, nanoseconds, and KiB.
3. Reproducible benchmark collection
   - validate source digests before compiling;
   - verify every original, Packforge, and UPX artifact before timing;
   - retain raw samples and compute deterministic medians;
   - record commit, OS, kernel, architecture, CPU, compiler versions, Packforge
     profile, UPX version, and UPX archive digest;
   - warm mode uses one unmeasured warm-up followed by 21 measured executions;
   - cold mode uses a dedicated runner, seven measured executions, and a complete
     filesystem page-cache reset before every execution.
4. Gate evaluator
   - correctness, reversibility, and determinism must pass for every fixture;
   - median Packforge/UPX size must be at most 1.05 for the release size gate;
   - median Packforge/UPX cold-start time must be below 1.00 unless Packforge has
     a documented median size win;
   - failed performance gates remain reportable data, not hidden or relabeled.
5. CI and publication
   - manual native benchmark jobs upload the JSON report, TSV view, raw samples,
     and environment text;
   - checked-in release baselines name their source workflow run and commit;
   - normal pull-request CI validates corpus metadata, schemas, and evaluator
     fixtures without treating noisy hosted timing as a merge gate.

## Implementation sequence

1. Add and validate the corpus manifest and report schema.
2. Split collection from rendering so raw samples remain authoritative.
3. Add dedicated warm and cold collectors with bounded iteration counts.
4. Add the deterministic gate evaluator and unit tests for passing, failing, and
   malformed reports.
5. Wire manual CI artifacts and execute a native baseline campaign.
6. Check in the baseline, update `BENCHMARKING.md`, and audit every exit criterion.

## Exit evidence

- The corpus manifest validates and every recorded source digest matches.
- A report validates against schema version 1 and contains raw samples for all
  twelve fixture/artifact combinations.
- Re-running aggregation over the same samples is byte-for-byte deterministic.
- The evaluator reports the current UPX comparison honestly, including failures.
- The linked native workflow completes successfully and uploads all documented
  artifacts.
- M0 status in `ROADMAP.md` changes to complete only after those facts are present.

M1 planning begins only after this audit passes.
